# Accessibility (a11y) baseline — Omni Runner portal

> **Finding:** [`L07-05`](./audit/findings/L07-05-portal-sem-acessibilidade-a11y-declarada.md)
> **Legal reference:** Lei Brasileira de Inclusão (Lei 13.146/2015)
> **Target standard:** WCAG 2.1 AA
> **CI guard:** `npm run audit:a11y-baseline`
> **Owner:** Portal squad.
> **Review cadence:** bi-annual + after any major UI refactor.

---

## 1. Target level

We commit to **WCAG 2.1 level AA** for the portal (`portal/src/app/**`).
Mobile (Flutter) has its own a11y backlog tracked separately — it
is **not** in scope for this document today.

WCAG 2.1 AA means:

- All interactive elements reachable by keyboard only.
- Colour-contrast ≥ 4.5:1 (text), 3:1 (large text, non-text UI).
- All images / icons convey meaning via `alt` or `aria-*`.
- Every form control has a `<label>` or `aria-label`.
- No flashing content above the 3 Hz threshold.
- Language declared (`<html lang="pt-BR">`) — already enforced
  by Next.js root layout.

## 2. Lint rules that MUST stay enabled

The portal ESLint config inherits `next/core-web-vitals`, which
transitively enables `eslint-plugin-jsx-a11y:recommended`. On top
of that, we pin the following rules to **error** (not warn) so a
regression fails CI:

| Rule                                            | Severity | Reason |
|-------------------------------------------------|----------|--------|
| `jsx-a11y/alt-text`                             | error    | Every `<img>` / icon must have alt. |
| `jsx-a11y/aria-props`                           | error    | Prevents invented aria attrs. |
| `jsx-a11y/aria-role`                            | error    | Role whitelist. |
| `jsx-a11y/aria-unsupported-elements`            | error    | No aria on inert elements. |
| `jsx-a11y/click-events-have-key-events`         | error    | Custody / swap actions MUST be keyboard-triggerable. |
| `jsx-a11y/heading-has-content`                  | error    | Empty headings break screen readers. |
| `jsx-a11y/label-has-associated-control`         | error    | Forms must associate labels with inputs. |
| `jsx-a11y/no-autofocus`                         | error    | Disorienting on SR. |
| `jsx-a11y/no-noninteractive-tabindex`           | error    | Prevent "focus black holes". |
| `jsx-a11y/role-has-required-aria-props`         | error    | Custom roles must declare their required props. |

Exceptions require a PR-review-visible `// eslint-disable-next-line`
**and** a corresponding row in §5 below.

## 3. High-risk patterns we check explicitly

The CI guard (`tools/audit/check-a11y-baseline.ts`) scans the
portal source for these regressions that ESLint alone can miss:

### 3.1 Icon-only buttons

```tsx
// ❌ Not accessible — icon only, no label
<button onClick={...}><TrashIcon /></button>

// ✅ Accessible — label on the button
<button onClick={...} aria-label="Excluir">
  <TrashIcon aria-hidden="true" />
</button>
```

### 3.2 Tables without `<caption>`

Every `<table>` that renders tabular data must carry a
`<caption>` (visible or SR-only) describing the columns.

### 3.3 Toasts / alerts without `role="alert"`

Non-modal toasts or banners that announce state changes must
carry `role="alert"` so assistive tech surfaces them.

### 3.4 Custody / swap action buttons

Buttons labelled by icon alone for "Distribuir", "Aceitar swap",
"Withdraw" MUST have an explicit `aria-label` OR visible text.
The audit finding (L07-05) called these out by name — the CI
guard tracks the filenames and refuses to let them regress to
icon-only.

## 4. Keyboard map

- **Tab / Shift-Tab** — focus moves through interactive elements in
  DOM order.
- **Enter / Space** on any `button` — activates.
- **Esc** on any modal — closes.
- **Arrow keys** on `role="radiogroup"` / `role="menu"` — cycle.
- All focus states must be visible (no `:focus { outline: none; }`
  without a replacement).

## 5. Exceptions log

All live exceptions. Every entry must link to a follow-up issue
that will remove the exception.

- **2026-04-21 — Ratchet baseline for `<table>` without `<caption>`** (38 files).
  Captured in [`tools/audit/a11y-baseline-allowlist.json`](../tools/audit/a11y-baseline-allowlist.json).
  New files are blocked by CI. Existing files will be remediated as
  part of follow-up `L07-05-tables-captions` — each remediation PR
  should also remove the corresponding entry from the allowlist.
  Adding entries is disallowed.

## 6. Runtime audit (axe-core)

In addition to static lint:

- `npm run audit:a11y-axe` — runs `@axe-core/cli` against a local
  portal dev server at `http://localhost:3000`. To be added as a
  follow-up (`L07-05-axe`) once the portal has a stable
  integration-test harness. Today, the **static** checks ship.

## 7. Manual audit cadence

- **Monthly**: portal squad walks the 8 most-trafficked pages with
  a screen reader (NVDA on Windows, VoiceOver on macOS). Record
  bugs as `L07-05-a11y-*` follow-ups.
- **Quarterly**: external audit against WCAG 2.1 AA checklist.
  First external audit is budgeted for Q3 2026; the report lives
  under `docs/compliance/a11y-audits/<YYYY-Qx>.md`.
