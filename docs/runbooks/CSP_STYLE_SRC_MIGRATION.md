# CSP `style-src 'unsafe-inline'` Migration Plan (L01-39)

## Status

* **Decision:** keep `style-src 'self' 'unsafe-inline'` for now.
* **Owner:** platform.
* **Re-evaluation trigger:** when we ship our first stylesheet that is
  NOT generated by Tailwind / shadcn-ui at compile time.

This document explains *why* the relaxation is currently acceptable
and what concrete steps a future contributor would take to drop the
`'unsafe-inline'` source.

---

## 1. Why we ship `'unsafe-inline'` today

The portal uses Tailwind CSS via `tailwindcss@3` and shadcn-ui
components. Tailwind compiles every utility class into a single
build-time `.css` artifact served from `_next/static/`, which is
already covered by `style-src 'self'`. However, three legitimate
runtime patterns still rely on inline `style="…"`:

| Pattern                           | Source                                | Lines (~) |
|-----------------------------------|---------------------------------------|----------:|
| Server-rendered chart bars        | `<div style={{ width: pct + "%" }}>`  | ~28       |
| Skeletons + radix-ui transitions  | `data-[state=open]` driven inline vars | ~14       |
| 3rd-party widgets (Stripe.js)     | injected by Stripe Elements           | n/a       |

CSS injection blast-radius is materially smaller than JS injection:
no script execution, no token exfiltration, no credential capture.
The OWASP CSP cheatsheet rates `style-src 'unsafe-inline'` as
"medium" risk vs. the "critical" rating for the JS analogue. Our
threat model accepts the medium rating until the patterns above are
removed.

JS-side `'unsafe-inline'` was already eliminated in L01-38 via the
nonce-based `script-src 'nonce-…' 'strict-dynamic'` rollout. The
nonce machinery (`portal/src/lib/security/csp.ts:generateNonce`) is
ready to host a `style-src 'nonce-…'` mirror when needed.

## 2. Migration plan (when triggered)

### 2.1 Inventory the offenders

```bash
rg -n "style=\\{\\{" portal/src --type tsx
rg -n "style=\"[^\"]" portal/src --type tsx
```

Categorise each hit:

* **Compile-time constant** → replace with a Tailwind utility class
  (e.g. `style={{ width: pct + "%" }}` becomes a CSS variable on the
  parent + `style="--w:${pct}%"` driving a `[width:var(--w)]`
  utility).
* **Per-request derived** → switch to a CSS variable injected via
  the response header (still nonce-protected) OR a small `<style
  nonce="…">` block emitted from the RSC.
* **3rd-party widget** → maintain an exception list; for Stripe
  specifically, the official guidance is to add `https://js.stripe.com`
  to `style-src` AND keep `'unsafe-inline'` because Stripe Elements
  injects per-iframe styles that can't be nonce-tagged.

### 2.2 Update `lib/security/csp.ts`

Add a `styleNonce` parameter to `buildCsp()` and emit:

```text
style-src 'self' 'nonce-${nonce}' https://js.stripe.com;
```

Drop `'unsafe-inline'`. The `style-src-attr` directive must also be
declared explicitly because the nonce source does NOT cover inline
`style="…"` attributes — those need either to be removed OR a
separate `style-src-attr 'unsafe-hashes' 'sha256-…'` for each
allowed attribute.

### 2.3 Plumb the nonce through RSCs

Mirror the `x-nonce` pattern from L01-38: middleware sets
`x-style-nonce` on the request headers; `app/layout.tsx` reads it via
`headers()` and renders `<style nonce={styleNonce}>` for every
runtime style block.

### 2.4 Browser smoke-test matrix

After the policy change, regression-test:

* Chrome 121+, Firefox 122+, Safari 17 — verify no console CSP
  violations on the dashboard, athlete detail, and clearing pages.
* Stripe Elements iframe (custody deposit) — verify the iframe
  renders correctly; if not, add the documented `style-src` exception.
* Mobile webview (omni_runner ↔ portal embedded views) — verify the
  policy is propagated and the webview renders correctly.

### 2.5 CSP report sink

The CSP report-uri (`/api/csp-report`, L10-05) already aggregates
`style-src` violations. Stage the rollout via Report-Only mode for 7
days before flipping to enforcing:

```text
Content-Security-Policy-Report-Only: style-src 'self' 'nonce-…';
Content-Security-Policy:             <existing relaxed policy>
```

If the report sink shows zero non-Stripe violations across one full
business cycle, switch the policies.

---

## 3. Acceptance criteria for closing this finding

1. `lib/security/csp.ts` no longer emits `'unsafe-inline'` in the
   `style-src` directive.
2. `npm run audit:csp-rollout` (planned guard) passes — checks the
   built CSP string for the absence of `'unsafe-inline'` in any
   `style-src*` directive.
3. CSP report sink shows < 1 violation per 10k page-views for 7
   consecutive days.
4. This document is updated to reflect the rollout date and removed
   from the "active" runbook list.
